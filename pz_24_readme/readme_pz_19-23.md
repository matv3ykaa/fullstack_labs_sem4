# Описание работы

Каждая работа посвящена отдельной теме: работе с реляционной и нереляционной СУБД, кэшированию через Redis, балансировке нагрузки и контейнеризации.

---

## ПЗ 19 (Работа с реляционными СУБД на примере PostgreSQL)

Реализовано серверное приложение на Node.js + Express с подключением к PostgreSQL через ORM Sequelize.

Модель `User` описана средствами Sequelize с опцией `underscored: true`, что автоматически приводит поля к snake_case в базе. Таблица синхронизируется с БД при старте через `sequelize.sync({ alter: true })`. Подключение настроено через connection string к локальному PostgreSQL.

Реализованы все требуемые эндпоинты:

| Адрес | Метод | Описание |
|---|---|---|
| `/api/users` | POST | Создание пользователя |
| `/api/users` | GET | Список пользователей |
| `/api/users/:id` | GET | Пользователь по ID |
| `/api/users/:id` | PATCH | Обновление пользователя |
| `/api/users/:id` | DELETE | Удаление пользователя |

Поля сущности: `id`, `first_name`, `last_name`, `age`, `created_at`, `updated_at`. Временны́е метки управляются автоматически Sequelize.

Стек: Node.js, Express, Sequelize, pg, PostgreSQL.

## ПЗ 20 (Работа с NoSQL СУБД на примере MongoDB)

Реализовано серверное приложение на Node.js + Express с подключением к MongoDB через Mongoose. Для изоляции окружения написан `docker-compose.yml`, поднимающий MongoDB-контейнер с health check и API-контейнер на `node:20-alpine`.

Схема `userSchema` описывает те же поля, что и в ПЗ 19. Временны́е метки `created_at` и `updated_at` хранятся в unix-формате (`Number`) и выставляются вручную через pre-save хук. Строка подключения читается из `.env` через `dotenv`.

Реализованы все требуемые эндпоинты с валидацией входных данных: проверка типов, диапазона возраста, непустых строк и корректности MongoDB ObjectId перед обращением к базе.

Отдельно реализован глобальный обработчик `unhandledRejection` и корректное завершение процесса при ошибке соединения.

Стек: Node.js, Express, Mongoose, MongoDB, Docker Compose.

## ПЗ 21 (Кэширование с использованием Redis)

Доработано приложение из ПЗ 11 (JWT-авторизация, RBAC, каталог товаров). Добавлено кэширование через Redis с использованием клиента `redis`.

Реализованы:
1. `cacheMiddleware(keyBuilder, ttl)` middleware, проверяющий наличие данных в Redis перед обращением к источнику; при попадании возвращает `{ source: "cache", data }`, иначе передаёт управление дальше и записывает ключи `req.cacheKey` / `req.cacheTTL` для последующего сохранения
2. `saveToCache(key, data, ttl)` сохранение результата в Redis с TTL
3. `invalidateUsersCache(userId?)` и `invalidateProductsCache(productId?)` инвалидация кэша при мутациях

Кэшируются маршруты `GET /api/users`, `GET /api/users/:id` (TTL 60 с), `GET /api/products`, `GET /api/products/:id` (TTL 600 с). При создании, обновлении или удалении записи соответствующие ключи сбрасываются.

Redis инициализируется асинхронно перед запуском сервера; при старте принудительно сбрасывается весь кэш пользователей и товаров.

Стек: Node.js, Express, Redis, bcrypt, jsonwebtoken, nanoid, swagger-jsdoc.

## ПЗ 22 (Балансировка нагрузки)

Реализована тестовая система балансировки нагрузки с двумя балансировщиками (Nginx и HAProxy, которые запускаются через Docker Compose с профилями).

Бэкенд представляет собой минимальный Express-сервер, отвечающий на `GET /` JSON-объектом с номером порта. Поднимаются три экземпляра (`backend1`, `backend2`, `backend3`) с health check через `wget`.

Nginx (`profiles: [nginx]`, порт 80) настроен с блоком `upstream` и параметрами отказоустойчивости `max_fails=2 fail_timeout=30s` для основных серверов; `backend3` выступает резервным (`backup`).

**HAProxy** (`profiles: [haproxy]`, порт 8080) настроен с алгоритмом `roundrobin`, активной проверкой через `httpchk GET /`, параметрами `inter 5s fall 3 rise 2` и резервным сервером.

Запуск нужного балансировщика:
```bash
docker compose --profile nginx up --build
docker compose --profile haproxy up --build
```

При остановке одного из основных бэкендов трафик автоматически перераспределяется на оставшиеся; при остановке обоих — поднимается резервный.

Стек: Node.js, Express, Nginx Alpine, HAProxy Alpine, Docker Compose.

## ПЗ 23 (Контейнеризация приложений с Docker)

Контейнеризация - это технология упаковки приложения вместе со всеми зависимостями в изолированный блок — контейнер. В отличие от виртуальной машины, контейнер использует ядро хостовой ОС и запускается за секунды, потребляя значительно меньше ресурсов.

Docker - это наиболее распространённая платформа для работы с контейнерами. Ключевые понятия: образ (неизменяемый шаблон со слоями), контейнер (запущенный экземпляр образа), Dockerfile (инструкции сборки), Docker Compose (декларативное описание многоконтейнерных приложений), Docker Hub (публичный реестр образов).

Важные концепции экосистемы:
1. Сети (Networks) - это изолированное пространство, внутри которого контейнеры обращаются друг к другу по имени сервиса как по имени хоста
2. Тома (Volumes) - это постоянное хранилище данных, монтируемое внутрь контейнера; без тома данные теряются при перезапуске
3. Переменные окружения - это механизм передачи конфигурации без изменения кода

Docker Compose позволяет описать весь стек сервисов в одном `docker-compose.yml` и управлять им одной командой. Поддерживает `depends_on`, `healthcheck`, `volumes`, `networks`, `profiles` и `deploy`.

Для оркестрации контейнеров в продакшене используются следующие решения:

1. Kubernetes (k8s) - стандарт индустрии. Предоставляет полный набор примитивов: Pod, Deployment, Service, Ingress, ConfigMap, Secret, HPA. Сложен в настройке, требует отдельного control plane (минимум 3 ноды для HA).

2. k3s - облегчённый дистрибутив Kubernetes от Rancher, упакованный в единый бинарник менее 100 МБ. Убраны устаревшие и облачные компоненты, встроен sqlite как хранилище по умолчанию. Идеален для edge, IoT и небольших кластеров. Полностью совместим с API Kubernetes.

3. k0s - ещё один минималистичный дистрибутив Kubernetes от Mirantis. Устанавливается как один бинарник без внешних зависимостей, поддерживает как single-node, так и HA-кластеры. Не требует root на worker-нодах.

Все три решения совместимы с одним и тем же `kubectl` и стандартными манифестами Kubernetes, поэтому переход между ними не требует переписывания конфигурации приложения.
